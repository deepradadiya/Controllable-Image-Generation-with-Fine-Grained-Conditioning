# Requirements Document

## Introduction

This feature implements the complete data pipeline for ControlNet training: downloading the "fusing/fill50k" dataset from HuggingFace, extracting three conditioning maps (depth, pose, edge), and verifying dataset integrity. The pipeline is optimized for Google Colab T4 GPU with checkpoint/resume support and produces training-ready paired data (source image + conditioning map + text prompt).

## Glossary

- **Pipeline**: The end-to-end data processing system comprising download, extraction, and verification stages
- **Conditioning_Map**: A derived image that encodes spatial information (depth, pose, or edges) used to guide ControlNet generation
- **Depth_Map**: A grayscale image where brighter pixels indicate closer proximity to the camera, generated by DPT (Dense Prediction Transformer)
- **Pose_Skeleton**: A stick-figure rendering of detected human body keypoints (shoulders, elbows, wrists, hips, knees), generated by DWPose
- **Edge_Map**: A binary image showing object outlines detected via rapid changes in pixel intensity, generated by Canny algorithm
- **DPT**: Dense Prediction Transformer — a Vision Transformer model trained to estimate per-pixel depth from a single RGB image
- **DWPose**: A whole-body pose estimation model that detects 18 body keypoints and renders colored stick figures
- **Canny**: An edge detection algorithm that identifies edges by detecting rapid changes in pixel intensity using gradient thresholds
- **Dataset_Downloader**: The module responsible for fetching the fusing/fill50k dataset from HuggingFace and saving it locally
- **Depth_Extractor**: The module responsible for generating depth maps from source images using Intel DPT-Large
- **Pose_Extractor**: The module responsible for generating pose skeleton images from source images using DWPose
- **Edge_Extractor**: The module responsible for generating edge maps from source images using OpenCV Canny
- **Dataset_Verifier**: The module responsible for checking completeness and integrity of all pipeline outputs
- **Checkpoint**: A saved progress marker that allows processing to resume from where it stopped after interruption
- **VRAM**: Video RAM available on the GPU (15GB on T4, ~13GB usable)
- **Batch_Processing**: Processing multiple images simultaneously to maximize GPU utilization

## Requirements

### Requirement 1: Dataset Download

**User Story:** As a researcher, I want to download the fusing/fill50k dataset from HuggingFace and save the first 5000 examples locally, so that I have paired images and prompts ready for conditioning map extraction.

#### Acceptance Criteria

1. WHEN the Dataset_Downloader is executed, THE Dataset_Downloader SHALL download the "fusing/fill50k" dataset from HuggingFace datasets library using the dataset's default split ordering
2. THE Dataset_Downloader SHALL select only the first 5000 examples from the dataset based on the dataset's default iteration order
3. THE Dataset_Downloader SHALL save source images to the `data/raw/images/` directory as individual PNG files named with zero-padded sequential indices (e.g., `00000.png` through `04999.png`)
4. THE Dataset_Downloader SHALL save all text prompts to `data/raw/prompts.json` as a JSON mapping of image filename (including extension) to prompt string
5. WHILE downloading, THE Dataset_Downloader SHALL display a tqdm progress bar showing download and save progress
6. WHEN an image is saved, THE Dataset_Downloader SHALL verify the image is not corrupted and has valid dimensions (width > 0 and height > 0)
7. IF an image fails validation, THEN THE Dataset_Downloader SHALL log the filename and error reason to `data/raw/bad_files.log` (one entry per line, format: filename followed by error reason) and continue processing remaining images
8. WHEN all images are processed, THE Dataset_Downloader SHALL print final statistics including total valid images saved, average width and average height in pixels across all valid images, and the prompt text of the first saved image
9. IF the HuggingFace dataset download fails due to a network error, THEN THE Dataset_Downloader SHALL retry the download up to 3 times with a 5-second delay between attempts before raising an error with a message indicating the failure reason

#### Correctness Properties

- **CP-1.1**: For all valid images saved, the count of entries in `prompts.json` SHALL equal the count of image files in `data/raw/images/`
- **CP-1.2**: For all entries in `prompts.json`, the referenced filename SHALL correspond to an existing file in `data/raw/images/`
- **CP-1.3**: The total count of valid images plus the count of entries in `bad_files.log` SHALL equal 5000

### Requirement 2: Depth Map Extraction

**User Story:** As a researcher, I want to extract depth maps from all source images using Intel DPT-Large, so that I have depth conditioning maps for ControlNet training.

#### Acceptance Criteria

1. WHEN the Depth_Extractor is executed, THE Depth_Extractor SHALL load the Intel DPT-Large model ("Intel/dpt-large") from HuggingFace
2. WHEN processing an image, THE Depth_Extractor SHALL resize the image to 512x512 pixels, run DPT inference, apply per-image min-max normalization to map the depth output to the 0-255 integer range, and save the result as a grayscale PNG
3. THE Depth_Extractor SHALL read source images from the `data/raw/images/` directory and save all depth maps to the `data/depth/` directory, preserving the original filename stem with a `.png` extension
4. THE Depth_Extractor SHALL process images in batches of 8 to fit within T4 GPU VRAM constraints
5. WHEN a previously processed image is encountered (output file already exists), THE Depth_Extractor SHALL skip that image to support checkpoint/resume after Colab resets
6. IF processing a single image fails due to a corrupted file or unsupported format, THEN THE Depth_Extractor SHALL log the filename and error reason, skip that image, and continue processing the remaining images
7. WHEN all images are processed, THE Depth_Extractor SHALL display 3 depth maps selected at equal intervals from the processed set, shown side by side with their original images using matplotlib
8. WHILE processing, THE Depth_Extractor SHALL display a tqdm progress bar showing batch processing progress

#### Correctness Properties

- **CP-2.1**: For all depth maps produced, the pixel values SHALL be in the range [0, 255] inclusive
- **CP-2.2**: For all depth maps produced, the image dimensions SHALL be exactly 512x512 pixels
- **CP-2.3**: For all source images in `data/raw/images/`, either a corresponding depth map exists in `data/depth/` or the image was logged as failed

### Requirement 3: Pose Skeleton Extraction

**User Story:** As a researcher, I want to extract pose skeletons from source images using DWPose, so that I have pose conditioning maps for ControlNet training.

#### Acceptance Criteria

1. WHEN the Pose_Extractor is executed, THE Pose_Extractor SHALL load the DWPose model from the controlnet_aux library
2. WHEN processing an image, THE Pose_Extractor SHALL run DWPose inference to detect human body keypoints and render a colored stick figure skeleton on a black background
3. WHEN no human keypoints are detected in an image, THE Pose_Extractor SHALL save a blank black image (all zeros, 512x512 pixels, RGB) as the pose map for that image
4. THE Pose_Extractor SHALL save all pose skeleton images to the `data/pose/` directory as PNG files, preserving the original filename stem with a `.png` extension
5. WHEN a previously processed image is encountered (output file already exists), THE Pose_Extractor SHALL skip that image to support checkpoint/resume after Colab resets
6. IF processing a single image fails due to a corrupted file or model error, THEN THE Pose_Extractor SHALL log the filename and error reason, skip that image, and continue processing the remaining images
7. WHEN all images are processed, THE Pose_Extractor SHALL display 3 example pose skeletons (preferring images where keypoints were detected) side by side with their original images using matplotlib
8. WHILE processing, THE Pose_Extractor SHALL display a tqdm progress bar showing processing progress

#### Correctness Properties

- **CP-3.1**: For all source images in `data/raw/images/`, exactly one corresponding file SHALL exist in `data/pose/` (either a rendered skeleton or a blank black image)
- **CP-3.2**: For all pose images produced, the image dimensions SHALL be 512x512 pixels with 3 color channels (RGB)
- **CP-3.3**: For images where no keypoints are detected, the pose output SHALL have all pixel values equal to 0

### Requirement 4: Edge Map Extraction

**User Story:** As a researcher, I want to extract edge maps from source images using OpenCV Canny, so that I have edge conditioning maps for ControlNet training.

#### Acceptance Criteria

1. WHEN the Edge_Extractor is executed, THE Edge_Extractor SHALL apply OpenCV Canny edge detection with low_threshold=100 and high_threshold=200 to each source image after converting it to grayscale
2. WHEN an edge map is produced, THE Edge_Extractor SHALL save it as a PNG file to the `data/edges/` directory using the same basename as the source image
3. THE Edge_Extractor SHALL execute entirely on CPU without requiring GPU resources
4. WHEN a previously processed image is encountered (output file already exists in `data/edges/`), THE Edge_Extractor SHALL skip that image to support checkpoint/resume
5. WHEN all images are processed, THE Edge_Extractor SHALL display 3 example edge maps selected from the first 3 successfully processed images, shown side by side with their original images using matplotlib in a single figure
6. WHILE processing, THE Edge_Extractor SHALL display a tqdm progress bar showing the number of images processed out of the total count
7. IF a source image cannot be read or is corrupted, THEN THE Edge_Extractor SHALL log a warning message indicating the failed filename and continue processing the remaining images
8. THE Edge_Extractor SHALL produce edge maps with the same pixel dimensions as the corresponding source images, saved as single-channel (grayscale) 8-bit PNG files with pixel values of 0 (non-edge) or 255 (edge)

#### Correctness Properties

- **CP-4.1**: For all edge maps produced, every pixel value SHALL be either 0 or 255 (binary output)
- **CP-4.2**: For all edge maps produced, the dimensions SHALL match the corresponding source image dimensions
- **CP-4.3**: For all source images in `data/raw/images/`, either a corresponding edge map exists in `data/edges/` or the image was logged as a warning

### Requirement 5: Dataset Verification

**User Story:** As a researcher, I want to verify that every sample has all required files (source image, depth map, pose skeleton, edge map, and prompt), so that I can confirm the dataset is complete and ready for training.

#### Acceptance Criteria

1. WHEN the Dataset_Verifier is executed, THE Dataset_Verifier SHALL enumerate all samples by listing filenames (without extension) in `data/raw/images/` and check that each sample has exactly 4 corresponding files: source image in `data/raw/images/`, depth map in `data/depth/`, pose skeleton in `data/pose/`, and edge map in `data/edges/`
2. WHEN the Dataset_Verifier is executed, THE Dataset_Verifier SHALL verify that every enumerated sample has a corresponding entry keyed by its filename in `data/raw/prompts.json`, and that the prompt value is a non-empty string of at least 5 characters
3. WHEN the Dataset_Verifier is executed, THE Dataset_Verifier SHALL verify that each conditioning map file (depth, pose, edge) is a loadable image file with a file size greater than 0 bytes, at least 256x256 pixels in resolution, and decodable without error by the image library
4. IF `data/raw/prompts.json` is missing or contains invalid JSON, THEN THE Dataset_Verifier SHALL report an error indicating the prompts file is unavailable and halt verification
5. IF any sample is missing one or more files or has files that fail the validity check in criterion 3, THEN THE Dataset_Verifier SHALL log to standard output the sample filename and list which specific files are missing or invalid
6. WHEN verification is complete, THE Dataset_Verifier SHALL report the total number of complete samples where all 4 files are present, pass validity checks, and have a valid prompt entry
7. WHEN verification is complete, THE Dataset_Verifier SHALL print a final summary message to standard output in the format: "X samples ready for training" where X is the count of complete valid samples

#### Correctness Properties

- **CP-5.1**: The reported count of complete samples SHALL equal the number of samples where all 4 files exist, are loadable, and have a valid prompt
- **CP-5.2**: The sum of complete samples plus flagged incomplete samples SHALL equal the total number of source images enumerated

### Requirement 6: Pipeline Documentation via Code Comments

**User Story:** As a researcher, I want each extraction script to contain a clear explanation of what a conditioning map is and how the specific extraction method works, so that I can understand the pipeline without external documentation.

#### Acceptance Criteria

1. THE Depth_Extractor script SHALL contain a module-level docstring as the first element in the file explaining that a depth map is a grayscale image where brighter pixels indicate closer proximity to the camera, and that DPT is a Vision Transformer trained to estimate depth from a single RGB image
2. THE Pose_Extractor script SHALL contain a module-level docstring as the first element in the file explaining that a pose skeleton is a stick figure showing human body keypoints (shoulders, elbows, wrists, hips, knees), and that DWPose detects body keypoints and draws colored stick figures representing limb connections
3. THE Edge_Extractor script SHALL contain a module-level docstring as the first element in the file explaining that an edge map is a binary image showing object outlines, and that Canny detects edges by finding rapid changes in pixel intensity
4. THE Edge_Extractor script SHALL include inline comments within 3 lines of the Canny threshold parameters explaining that low_threshold=100 means edges with gradient magnitude below 100 are discarded, and high_threshold=200 means edges with gradient magnitude above 200 are always kept
5. WHEN a module-level docstring is present in any extractor script, THE docstring SHALL contain no fewer than 30 words explaining the conditioning map type and the extraction method used
